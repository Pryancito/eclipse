# hunter — Adversarial hardening report

Generated by a multi-agent red-team workflow (recon → per-lens attack → adversarial verification → completeness critic → hardening design) against the hunter security subsystem as implemented on this branch.

**Result:** 20 confirmed findings, 7 uncertain, 10 additional critic gaps.

## Executive summary

hunter is structurally sound as an LSM-style skeleton but ships effectively disabled and bypassable by construction. The single most damaging fact is that no kernel code ever raises any domain out of its default mode or registers any policy: SYSCALL_MODE=Enforce is inert because GLOBAL_POLICIES is always empty (every (pid,syscall) hits the `None => Ok(())` default-permissive arm at policy.rs:158), and WX_MODE/EXEC_MODE default to Report so every W+X mapping and every /tmp exec is logged-and-allowed. Even if modes were flipped, four design defects defeat enforcement: (a) W^X is per-call-conjunction only, so mmap(W)+mprotect(X) two-step is invisible (no per-region history); (b) file-backed mmap is force-mapped MMUFlags::RXW regardless of requested prot (vm.rs:139), giving a writable+executable ceiling that a later mprotect(X) always satisfies; (c) the exec gate is a 4-byte magic compare plus substring path matching on the *unresolved* user string, defeated by /proc/self/fd/N magic-links, symlinks, relative paths, and a second weaker loader gate (linux.rs:165) that reads only 4 bytes against args[0]; (d) the dynamic-linker/shebang interpreter is loaded with zero hunter checks. The forensic log is a 512-entry ring with silent eviction, so any attacker floods it to erase evidence, and the only un-suppressable signal (DROPPED) cannot be tied to which events were lost. State is keyed solely on pid, never re-keyed at fork/exec, and freed only via exit_group, so plain _exit/signal-kill leak BTreeMap entries unboundedly (a self-DoS that also slows the syscall hot path under a single Mutex). Finally the whole control plane is unauthenticated, unlogged, and the clock is an unauthenticated transmute'd function pointer that gates both forensics and the IDS window.

The hardening plan below is ordered to (1) make enforcement *reachable* and *correct* without breaking boot/userspace by keeping conservative defaults, (2) close the W^X and exec structural bypasses, (3) make the forensic record tamper-evident and the IDS evasion-resistant, and (4) lock down the control plane and clock. Every item maps to a concrete module change and notes userspace-breakage risk. The guiding constraint is preserved throughout: enforcement only bites when explicitly opted into (a registered policy / a raised mode), so the default boot path stays permissive and cannot regress userspace.

## Confirmed findings (deduped, by corrected severity)

### `WX-1` [HIGH] W^X defaults to Report mode and no process ever flips it to Enforce — every W+X mapping is allowed
- **Lens:** W^X memory evasion
- **Location:** hunter/src/policy.rs:74 (WX_MODE=1/Report); hunter/src/lib.rs:165-176 (wx_check Report branch returns true); linux-syscall/src/vm.rs:106-112, 285-291
- **Bypass:** Attacker calls mmap(prot=PROT_READ|PROT_WRITE|PROT_EXEC, MAP_ANONYMOUS) directly, writes shellcode into the page, and executes it in-place. wx_check sees prot_write && prot_exec, reads wx_mode()==Mode::Report, records a Warning audit line, and returns true (lib.rs:167-176); sys_mmap then maps the anonymous VMO with prot.to_flags() = USER|READ|WRITE|EXECUTE (vm.rs:124, vm.rs:368-384). The page is genuinely RWX. Because set_wx_mode() is never called anywhere in the kernel (only in-crate tests), WX_MODE stays at its Report default forever, so the Enforce branch (lib.rs:177-188, the only branch that returns false → EACCES) is dead code in production. W^X is effectively disabled by default.
- **Exploitability:** Trivial and reliable. Preconditions: (a) userspace can issue a normal mmap syscall with PROT_READ|PROT_WRITE|PROT_EXEC and MAP_ANONYMOUS — standard Linux ABI, and no syscall whitelist is registered for the process so policy.None→Ok permits mmap (policy.rs:158); (b) default boot configuration, which is the only reachable configuration since no code enables Enforce. The attacker writes shellcode into the returned RWX page and jumps to it; hunter merely logs a Warning audit line and allows it. No race, no info leak, no privilege needed beyond running unprivileged code. Severity lowered from critical to high because this is the documented default posture of a report-mode mitigation (defense-evasion / hardening gap) rather than a memory-safety or privilege-escalation primitive on its own — it removes an exploit-mitigation layer rather than directly granting code execution that wasn't otherwise reachable.

### `WX-2` [HIGH] Two-step mmap-W-then-mprotect-X defeats the per-call check even if Enforce were on
- **Lens:** W^X memory evasion
- **Location:** hunter/src/lib.rs:161-164 (wx_check only fires when BOTH write && exec in the same call); linux-syscall/src/vm.rs:106-112 (mmap), 285-291 (mprotect)
- **Bypass:** The check is purely per-call and only triggers on the conjunction prot_write && prot_exec within one syscall (lib.rs:162: `if !(prot_write && prot_exec) { return true; }`). Attacker: (1) mmap(PROT_READ|PROT_WRITE) — write-only, exec bit clear → check passes trivially; write shellcode. (2) mprotect(PROT_READ|PROT_EXEC) — exec-only, write bit clear → check passes trivially. The region is now executable and still contains attacker-controlled bytes. At no single instant does any one call present both bits, so even with WX_MODE=Enforce hunter never sees a violation. This is the canonical JIT/shellcode flip and hunter has no notion of a region's history — it has no per-region W-then-X state, only per-call boolean flags.
- **Exploitability:** Preconditions are minimal and fully under attacker control from unprivileged userspace: the attacker needs only to already be executing code in the process (the universal precondition for any W^X / shellcode-injection defense — W^X exists precisely to stop a memory-corruption foothold from escalating to arbitrary native-code execution). No special fd, path, or capability is required. The attacker issues mmap(...,PROT_READ|PROT_WRITE,MAP_ANONYMOUS,...), copies shellcode in, then mprotect(addr,len,PROT_READ|PROT_EXEC), then jumps to addr. Both syscalls pass hunter unconditionally because neither presents write&&exec simultaneously, and the underlying VMAR's RXW ceiling permits the protection change for both anonymous and file-backed maps, so the page table really becomes executable. Works regardless of WX mode (Off/Report/Enforce) since the guard never fires. The only thing hunter would catch is a single-call mmap/mprotect that requests W+X at once — exactly the case a real attacker avoids. Severity high (not critical) because hunter is an in-kernel hardening/IDS layer, not the sole barrier, and the attack still presupposes an existing code-exec foothold; but it completely defeats the stated W^X-evasion defense for its primary threat model.

### `WX-3` [HIGH] File-backed mmap is mapped with an RXW permission ceiling, enabling a later mprotect to EXECUTE on writable file pages
- **Lens:** W^X memory evasion
- **Location:** linux-syscall/src/vm.rs:134-143 (map_ext called with permissions=MMUFlags::RXW); zircon-object/src/vm/vmar.rs:210, 931-933 (is_valid_mapping_flags ceiling check), 935-948 (protect)
- **Bypass:** Every file-backed mmap passes permissions=MMUFlags::RXW as the mapping's permission ceiling (vm.rs:139), independent of the requested prot. The per-page protection ceiling check is is_valid_mapping_flags: `self.permissions.contains(flags & RXW)` (vmar.rs:932). Since the ceiling is RXW (= READ|WRITE|EXECUTE, defs.rs:24), it contains ANY combination, so a later mprotect(PROT_READ|PROT_EXEC) — or even PROT_READ|PROT_WRITE|PROT_EXEC — on a file-backed region always satisfies the kernel-side ceiling and actually flips the page flags to executable (vmar.rs:935-948). Combined with WX-2, an attacker mmaps a file (e.g. a writable scratch file or even MAP_PRIVATE of any file) PROT_READ|PROT_WRITE, modifies the private copy, then mprotect()s it executable; the RXW ceiling guarantees the kernel permits the exec transition. hunter's W^X check is the only thing standing in the way and it is in Report mode (WX-1).
- **Exploitability:** Realistic and low-precondition. Attacker needs only the ability to run code in a userspace process (the threat model W^X is meant to constrain) and issue ordinary mmap/mprotect syscalls — no special privilege, no filesystem write (MAP_PRIVATE copy-on-write of any readable file, or a plain anonymous mapping, both work). Sequence: mmap a region PROT_READ|PROT_WRITE, write shellcode into the private/anon pages, then mprotect(PROT_READ|PROT_EXEC). Because the Zircon-side ceiling is RXW for every mapping and hunter's wx_check ignores any prot lacking the write bit, the exec transition succeeds and the page table is updated to executable (vmar.rs:943-946). With hunter at its default Report mode, even the single-shot RWX mmap/mprotect works. The only thing that would blunt this is set_wx_mode(Enforce) AND restricting to the RWX-in-one-call case, which the RW->RX two-step trivially sidesteps. Caveat: this is the standard JIT/dynamic-linker pattern, so any real-world enforcement would also break legitimate runtimes — which is exactly why the project ships wx in Report mode by default.

### `WX-1` [HIGH] W^X is audit-only by default and structurally bypassable even if enforced
- **Lens:** threat-model & info-leak
- **Location:** hunter/src/policy.rs:74; hunter/src/lib.rs:161-189; linux-syscall/src/vm.rs:139,298-306
- **Bypass:** Several independent paths defeat W^X. (1) WX_MODE defaults to Report (policy.rs:74) and is never flipped from userspace, so W+X mmap/mprotect is logged and allowed. (2) Even in Enforce, wx_check only fires when BOTH write and exec bits are set in the same call (lib.rs:162); the attacker maps writable, writes shellcode, then mprotect(PROT_EXEC) with the write bit cleared -- neither call has both bits, no violation. (3) File-backed mmap is mapped MMUFlags::RXW unconditionally regardless of requested prot (vm.rs:139), yielding a writable+executable file mapping behind hunter's back. (4) sys_mprotect's overlap fallback returns Ok(0) silently on protect() error (vm.rs:298-306), so segments the kernel mapped with liberal perms stay executable while the syscall reports success. (5) brk-grown heap is never routed through any hunter hook.
- **Exploitability:** Default config (no operator action): trivial. Any unprivileged process issues a single mmap(..., PROT_READ|PROT_WRITE|PROT_EXEC, MAP_ANONYMOUS, ...); because WX_MODE defaults to Report, the call succeeds and yields a W+X anonymous region the attacker writes shellcode into and executes. No special preconditions, no race. Even if an operator hardens WX to Enforce, the write-then-exec split (mmap RW, write payload, mprotect RX) defeats it with two ordinary syscalls and no race, because hunter only inspects per-call write&exec bit pairs and the Zircon mapping's RXW ceiling permits the RX transition. The file-mmap-RXW and brk vectors do not actually produce attacker-controlled W+X memory, so they add nothing. Severity remains high: a single self-inflicted-process W^X bypass is exactly the primitive W^X exists to deny (post-exploitation code execution from a writable buffer), it needs only unprivileged syscalls, but it does not by itself cross a privilege boundary, so not critical.

### `SYS-1` [MEDIUM] Syscall whitelist is globally inert: no process is ever given a policy, so every syscall hits the default-permissive branch
- **Lens:** syscall-filter bypass
- **Location:** hunter/src/policy.rs:148-160 (esp. :158); hunter/src/lib.rs:76-97; linux-syscall/src/lib.rs:130; zCore/src/main.rs:62-66
- **Bypass:** An attacker needs to do nothing. The only function that can populate the whitelist registry is policy::register_policy (policy.rs:128), and a repo-wide search shows it is invoked exclusively inside the hunter crate's own unit tests (lib.rs:238) and docs — never from kernel boot (zCore/src/main.rs only calls set_time_source + init at :65-66), never from execve, fork, or any process-creation path. GLOBAL_POLICIES is therefore always empty, so for every (pid, syscall) the match in is_syscall_allowed falls to `None => Ok(())` (policy.rs:158). check_syscall returns Ok and linux-syscall/src/lib.rs:130 never returns -EPERM. Despite SYSCALL_MODE defaulting to Enforce (=2, policy.rs:73), the 'seccomp' filter blocks zero syscalls in the running system — any process can issue any syscall (ptrace, init_module, bpf, mount, kexec, execve of anything). active_policy_count() is structurally always 0.
- **Exploitability:** No attacker action required and no privilege boundary is crossed by the defect itself: the syscall-whitelist domain simply provides zero confinement to every process. Precondition to "exploit": run any process and issue any syscall (ptrace, init_module, bpf, mount, kexec, etc.) — it will never be denied by hunter's seccomp domain regardless of syscall number, because no pid ever has a policy. This is real and trivially demonstrable. However, it only nullifies a confinement feature that was never enabled/populated, not an active filter; it does not by itself grant new privileges beyond whatever the rest of the kernel's permission checks already allow. The W^X and exec-path domains remain operative, so this does not blanket-disable hunter.

### `SYS-2` [MEDIUM] task_exit hooked only in exit_group, not plain exit — enables PID-reuse policy/state laundering and unbounded BTreeMap DoS
- **Lens:** syscall-filter bypass
- **Location:** linux-syscall/src/task.rs:634-638 (sys_exit, no hook) vs task.rs:645-653 (sys_exit_group calls hunter::task_exit); hunter/src/lib.rs:193-196; hunter/src/heuristics.rs:196-198
- **Bypass:** hunter::task_exit (which calls remove_policy + heuristics::forget) is wired into ONLY sys_exit_group (task.rs:650). sys_exit (task.rs:634, the raw _exit) and all signal-kill / thread-teardown exit paths never call it. Two abuses: (1) PID-reuse laundering — even if a future deployment did register a per-pid whitelist or accumulate anomaly counters, a process that terminates via _exit (or is killed) leaves its GLOBAL_POLICIES/PROC_STATS entry behind; because all hunter state is keyed solely on zircon_process().id() (lib.rs:130, heuristics.rs:151), a recycled PID inherits stale policy or, conversely, a process can cheaply shed accumulated anomaly state by exiting via _exit and being respawned. (2) State-leak DoS — PROC_STATS gains an entry on the first syscall of every process (heuristics.rs:151) and is never freed for processes that don't call exit_group; an attacker spawning many short-lived processes that _exit grows the locked BTreeMap without bound, adding lock contention on the syscall hot path and consuming kernel heap.
- **Exploitability:** No special privilege needed. Any unprivileged process that issues at least one syscall (which inserts its PROC_STATS entry, heuristics.rs:151) and then terminates by any path other than sys_exit_group leaves that entry leaked forever. The non-exit_group paths are confirmed in code: sys_exit/exit_linux -> self.exit() (thread.rs:125, single-threaded _exit), signal kill (signal.rs:161/163/200/202), exception/SYSCALL_KILL (exception.rs:401), pidfd kill (pidfd.rs:57), loader (linux.rs:325/394) — none call hunter::task_exit; the only caller is sys_exit_group (task.rs:650). An attacker spawns many short-lived processes that each make one syscall then _exit (or get signal-killed) to grow GLOBAL/PROC_STATS without bound, adding global-lock contention on the syscall hot path (heuristics.rs:127-165, lock held per-syscall) and consuming kernel heap. Practical preconditions: ability to spawn processes (normal for any shell/user) and default config (ANOMALY_ENABLED defaults true, heuristics.rs:30). The bound on impact: musl/glibc normally route process termination through exit_group, so the leak is driven by threads' _exit, signal/abnormal termination, and an attacker deliberately calling raw _exit(2) — all readily reachable.

### `SYS-4` [MEDIUM] W^X is bypassable by construction: only user-supplied prot flags are checked, file-backed mmap is force-mapped RXW, and default mode is Report
- **Lens:** syscall-filter bypass
- **Location:** linux-syscall/src/vm.rs:106-112, :134-145 (RXW at :139); linux-syscall/src/vm.rs:284-307 (:298-306 fallback); hunter/src/lib.rs:161-189; hunter/src/policy.rs:74
- **Bypass:** Three independent escapes from W^X enforcement. (1) Default WX_MODE = Report (policy.rs:74) and no kernel code ever calls set_wx_mode (see SYS-1 grep), so even a literal mmap(PROT_WRITE|PROT_EXEC) is only logged and allowed (lib.rs:167-176). (2) Even if flipped to Enforce, check_mmap only inspects the user-passed prot bits (vm.rs:108-109); the non-anonymous path then maps the VMO with hardcoded MMUFlags::RXW regardless of prot (vm.rs:139), so a file-backed mapping requested as PROT_READ|PROT_EXEC (passes the W&&X gate, lib.rs:162) is installed as readable-writable-executable in the page tables — writable+executable memory with no W^X event recorded. (3) The classic two-step is unguarded: map a file/anon region writable, write shellcode, then mprotect it executable. mprotect's check only fires when WRITE&&EXEC are BOTH set in the same call (lib.rs:162); requesting PROT_READ|PROT_EXEC alone passes, and on overlap errors vmar.protect failures are swallowed as Ok(0) (vm.rs:298-306), so the caller can't even tell enforcement happened. brk-grown heap (sys_brk) is never routed through any W^X hook at all.
- **Exploitability:** Realistic preconditions: the attacker must already be able to issue mmap/mprotect in a target process (i.e. already have a code-exec or scripting primitive); W^X is a defense-in-depth layer, not a first-line control. Given that, escape (1) is trivial and always available as shipped: WX_MODE=Report by default and nothing flips it, so any mmap(PROT_WRITE|PROT_EXEC) succeeds (only a /proc/hunter Warning is logged). Even if an operator manually calls set_wx_mode(Enforce), escape (3) — mmap RW, write shellcode, mprotect R|X — still works because hunter only blocks simultaneous W&X in a single call; this requires no special conditions. Escape (2) (file-backed RWX in page tables) is NOT exploitable: the page tables receive RX, not RXW, so it cannot be used to obtain writable+executable memory. Overall: the off-by-default W^X is a real but expected hardening limitation, exploitable only by an attacker who already has the ability to call these syscalls.

### `SYS-5` [MEDIUM] execve ELF/path gate is trivially satisfiable: 4-byte magic only, substring path match, default Report, and 64-byte read
- **Lens:** syscall-filter bypass
- **Location:** hunter/src/lib.rs:106-146; hunter/src/policy.rs:174-180; linux-syscall/src/task.rs:546-552; hunter/src/policy.rs:75
- **Bypass:** check_elf_binary 'integrity' is just the 4-byte \x7fELF magic comparison (lib.rs:108) over the first 64 bytes read at task.rs:547-548 — no signature, hash, section, or interpreter validation. Any attacker-controlled blob beginning with the ELF magic passes; a malicious binary is indistinguishable from a trusted one. The path policy (policy.rs:174-180) is pure string matching: substring '../' and starts_with on the fixed prefixes /tmp/, /var/tmp/, /dev/shm/. It is defeated by symlinks, bind/hardlinks, or simply executing from any other writable location (e.g. /run/, /home/, a mounted volume) since add_untrusted_exec_prefix is never called to extend the list (SYS-1). Most importantly EXEC_MODE defaults to Report (policy.rs:75) with no kernel caller flipping it, so even a path that DOES match (e.g. /tmp/evil) is only logged and execve proceeds (lib.rs:135-143, returns true). The kernel honors the verdict (task.rs:549-551 returns EACCES on false) but in the shipped default the verdict is always true for any ELF-magic file.
- **Exploitability:** In the shipped default (EXEC_MODE=Report) there are essentially no preconditions: any process able to write a file starting with the 4-byte ELF magic and execve it runs unimpeded; the exec-path policy is log-only, so even /tmp/evil executes. The attacker needs only normal execve capability over an attacker-staged file (the usual situation after gaining any code-exec foothold). If an operator has opted into Enforce mode (which no production code does — requires manual policy::set_exec_mode(Mode::Enforce)), the bypass still holds with trivial effort: place the binary outside /tmp,/var/tmp,/dev/shm (e.g. /run, /home, any mount) or reach a listed dir via a symlink/bind-mount/hardlink whose user-supplied path string lacks "../" and the listed prefixes, since hunter checks the unresolved path_str. No kernel exploit, race, or privilege beyond ordinary file-write+exec is required. This is best characterized as a weak/default-off control rather than a control that is actively circumvented, which is why medium (not high) is appropriate.

### `ELF-2` [MEDIUM] /proc/self/fd/N magic-link defeats exec-path policy entirely (even in Enforce)
- **Lens:** ELF integrity & exec-path bypass
- **Location:** linux-object/src/fs/mod.rs:1170-1196 (magic-link resolution) vs hunter/src/policy.rs:174-180 (string-only path check)
- **Bypass:** hunter's path policy is pure string matching on the literal execve argument (policy::is_untrusted_exec_path: substring "../" or starts_with /tmp//var/tmp//dev/shm/). But sys_execve resolves the path through linux_process().lookup_inode (task.rs:540), and lookup_inode_at special-cases /proc/self/fd/N: with follow=true it returns the *open file's inode directly* (mod.rs:1186-1194, "runs the file, not the symlink's path text"). Attacker: open("/tmp/evil") to get fd N (open is not hooked at all), then execve("/proc/self/fd/N"). hunter sees path "/proc/self/fd/N" — no "../", matches no untrusted prefix — so is_untrusted_exec_path returns false and the exec is ALLOWED even with exec_mode=Enforce. The bytes hunter reads at offset 0 are the real /tmp/evil bytes, so even the magic check is satisfied legitimately. The world-writable-location policy is completely bypassed.
- **Exploitability:** Preconditions: (1) operator has opted EXEC_MODE into Enforce — otherwise the exec-path policy is in default Report mode and the world-writable-path exec is already permitted with no bypass needed; (2) attacker can already execute code in a process (to issue open + execve) and can place an ELF in a world-writable dir. Given those, the bypass is trivial and 100% reliable: open("/tmp/evil") -> fd N (unhooked), then execve("/proc/self/fd/N"). No race, no special privilege. Impact is bounded because this policy is an opt-in, string-only, defense-in-depth control that is weak by construction (e.g. also evadable by hardlinking/copying into a trusted path if writable, or via the resolved-but-still-string-checked nature of the matcher); it is not a primary isolation boundary. Hence medium rather than high.

### `ELF-3` [MEDIUM] Symlink and relative-path divergence: path string checked, symlink target loaded
- **Lens:** ELF integrity & exec-path bypass
- **Location:** linux-object/src/fs/mod.rs:1198-1216 / 1223-1225 (follow=true symlink resolution) vs hunter/src/policy.rs:174-180
- **Bypass:** lookup_inode resolves with follow=true (mod.rs:1224) up to FOLLOW_MAX_DEPTH symlinks, but hunter checks only the unresolved user string. Two concrete bypasses, both effective even in Enforce: (a) Symlink: ln -s /tmp/evil /home/user/launch; execve("/home/user/launch") — string starts with /home/, no "../", policy says trusted; the inode loaded is /tmp/evil. (b) Relative path: with cwd=/, execve("tmp/evil") or execve("./tmp/evil") resolves to /tmp/evil but the string does not start_with "/tmp/" (it's "tmp/evil" or "./tmp/evil") and contains no "../", so is_untrusted_exec_path returns false. The untrusted-prefix list is anchored to absolute leading-slash forms only, so any non-canonical spelling that resolves to the same file evades it.
- **Exploitability:** Requires the exec-path domain to be in Enforce mode (non-default; default is Report). Given that, exploitation is trivial and needs no privilege escalation: (a) symlink variant — attacker drops a payload at an untrusted location (e.g. /tmp/evil) and creates a symlink to it from any path the policy treats as trusted, then execs the symlink path; a single hop suffices (FOLLOW_MAX_DEPTH=1). (b) relative-path variant — attacker sets cwd appropriately (e.g. /) and execs a non-canonical spelling ("tmp/evil", "./tmp/evil") that resolves to the blocked file. Both run the intended /tmp binary while hunter logs/permits a benign-looking string. Downgraded from high to medium because the impact is confined to one opt-in, explicitly best-effort advisory domain rather than the kernel's primary exec authorization, and is inert in the default (Report) configuration. The correct fix is to canonicalize/resolve the path (and compare the resolved target's location) before passing it to hunter, rather than checking the raw user string.

### `IDS-1` [MEDIUM] 512-entry log ring is trivially flooded to evict attack evidence (only DROPPED counter survives)
- **Lens:** IDS evasion & self-DoS
- **Location:** hunter/src/event_log.rs:21,98-107,196-216; hunter/src/lib.rs:200-224
- **Bypass:** The forensic record is a 512-slot ring (GLOBAL_LOG = IntrusionLog::new(512)). push() overwrites the oldest entry in place once full and only bumps a DROPPED counter (event_log.rs:103-105). render()/get_entries() returns ONLY the surviving 512 entries; the evicted LogEntry contents are gone forever. An attacker performs the real attack (e.g. a watched ptrace/init_module/bpf, a W+X mmap in Report mode, an exec from /tmp), then immediately issues >=512 benign events that hunter itself records — e.g. spam any sensitive syscall from heuristics::classify (each call records a NOTICE/WARN 'WATCH' entry, heuristics.rs:130-138) or any reported syscall/WX warning. After 512 such events the original CRIT/WARN line has been overwritten and /proc/hunter shows only the attacker's noise plus a bumped 'dropped' number. Since seq gaps are the only hint (event_log.rs:64), and the human-readable render gives no indication WHICH events were lost, the responder cannot recover what the attacker did. The flood costs ~512 cheap syscalls and is itself indistinguishable from the noise.
- **Exploitability:** Realistic preconditions are modest but not trivial. The attacker needs to (a) already have performed the real attack that hunter recorded, and (b) be able to issue >=512 syscalls that hunter records as events. The cheapest path is spamming any classify()-matched sensitive syscall (ptrace, setuid, bpf, etc.) from a process with no registered seccomp whitelist (default-permissive), since those WATCH records happen regardless of mode. ~512 cheap syscalls suffice and run in O(1) each. No special privilege beyond making syscalls is required. BUT: the surviving monotonic counters (criticals/warnings/blocked/dropped) still reveal that an incident of a given class occurred and that eviction happened, and the 512 identical flood entries are themselves conspicuous, so the technique destroys per-event detail rather than achieving stealthy evasion. Net: easy to trigger, but degraded forensics rather than a clean bypass, and the responder retains strong tells.

### `IDS-2` [MEDIUM] sys_exit (and all non-exit_group teardown) never frees hunter state — unbounded PROC_STATS / GLOBAL_POLICIES growth = kernel memory DoS
- **Lens:** IDS evasion & self-DoS
- **Location:** linux-syscall/src/task.rs:634-638 (sys_exit) vs 645-652 (sys_exit_group); hunter/src/lib.rs:193-196; hunter/src/heuristics.rs:151,196-198
- **Bypass:** hunter::task_exit (which calls policy::remove_policy + heuristics::forget) is wired into ONLY sys_exit_group (task.rs:650). sys_exit (task.rs:634-638) terminates the thread/process without ever calling task_exit, as do signal kills / thread teardown. Every process that issues at least one syscall gets a PROC_STATS BTreeMap entry created lazily (heuristics.rs:151, entry(pid).or_insert_with). A process that exits via raw _exit / exit(2) (glibc's _exit, and any process killed by signal) leaves its ProcStat entry — and any registered whitelist — in the global BTreeMaps forever. An attacker spawns and _exit()s millions of short-lived processes (each touching one syscall to seed PROC_STATS); the BTreeMap grows without bound, consuming kernel heap until OOM. This is a self-DoS of the whole kernel driven entirely from unprivileged userspace, and it also blinds the IDS: the leaked entries accumulate under a single Mutex that every syscall must take.
- **Exploitability:** Reachable from unprivileged userspace under the default config (ANOMALY_ENABLED = true). Each process that issues at least one allowed syscall creates a PROC_STATS BTreeMap entry (heuristics.rs:151, via check_syscall -> on_syscall, lib.rs:94). That entry is freed ONLY by hunter::task_exit, which is wired into exactly one place: sys_exit_group (task.rs:650), confirmed by a tree-wide grep. Any other teardown leaves the entry forever: raw SYS_exit / _exit for a single-threaded process (task.rs:634 -> thread.exit_linux -> remove_thread -> Process::terminate, process.rs:294, which calls NO hunter hook), signal kills (kill/terminate path), and non-last-thread exits. zCore commonly runs musl, whose raw _exit and signal-killed processes do not route through exit_group. PROC_STATS is keyed by the monotonic zircon KoID (not a recycled small pid), so leaked entries are never overwritten/coalesced and accumulate one-per-distinct-process. An attacker spawning a long stream of short-lived processes that each touch one syscall then _exit() (or are signalled) grows the map without bound under a global Mutex taken on every syscall, eventually pressuring the kernel heap (self-DoS) and adding lock contention.

### `SYS-1` [MEDIUM] Seccomp whitelist is inert: no policy is ever registered, so every syscall hits the default-permissive branch
- **Lens:** threat-model & info-leak
- **Location:** hunter/src/policy.rs:148-159; linux-syscall/src/lib.rs:130
- **Bypass:** An attacker process simply issues any syscall it wants. The dispatch chokepoint calls hunter::check_syscall(pid, num, &args) (lib.rs:130) which calls policy::is_syscall_allowed. With SYSCALL_MODE defaulting to Enforce (policy.rs:73) the Off short-circuit is skipped, then GLOBAL_POLICIES.get(&pid) returns None for every real process because register_policy is never invoked anywhere in the kernel (only in-crate tests). The None arm returns Ok(()) (policy.rs:158). So the 'enforcing seccomp' layer blocks literally nothing for any process. There is no userspace interface (no prctl/ioctl/syscall) to ever populate a whitelist, so the gate is permanently open by construction, not misconfiguration.
- **Exploitability:** Trivial preconditions: any process making any syscall is already past the gate. No race, no privilege, no special timing, no crafted input needed — the default-permissive None arm is hit for every real pid on every syscall because no whitelist is ever registered. So the "bypass" requires literally nothing of the attacker. However, "exploit" overstates impact: bypassing an inert filter grants no capability beyond what a process without any seccomp layer already has. The realistic harm is false assurance — an operator reading /proc/hunter sees syscall=enforce and may believe syscall filtering is active when it can never block anything. There is also no in-tree path for a defender to fix this at runtime (no API surfaced to userspace to call register_policy / set_syscall_mode), so the layer is permanently open until code is added to wire it.

### `EVADE-1` [MEDIUM] 512-entry ring with silent eviction lets an attacker flush all forensic evidence
- **Lens:** threat-model & info-leak
- **Location:** hunter/src/event_log.rs:21,98-107; hunter/src/heuristics.rs:130-138
- **Bypass:** GLOBAL_LOG is a fixed 512-entry ring (event_log.rs:21). When full, push() overwrites the oldest entry, only incrementing a DROPPED counter (event_log.rs:101-105). After a noisy attack (e.g. a watched ptrace=101 / init_module=175 / bpf=321 that records an audit event), the attacker emits >512 additional logged events to evict every record of the attack. Generators are trivial: each classify() hit on the sensitive-syscall WATCH path records an event with no de-duplication or rate limit (heuristics.rs:130-138 -- only the flood/fork alerts are de-duped, not the per-syscall WATCH records). /proc/hunter then shows only attacker-chosen filler; high-severity Critical/Warning entries are unrecoverable. There is no persistence, no off-box sink, and no protection that pins high-severity entries.
- **Exploitability:** Realistic and cheap as written. classify() matches many syscalls that the surrounding kernel allows by default (ptrace, setuid/setgid, bpf, mount, unshare, keyctl, etc.) and the WATCH record fires unconditionally after the policy check with no de-dup. A simple userspace loop calling e.g. ptrace(PTRACE_TRACEME)/setuid(getuid()) >512 times overwrites the entire ring in well under a second; staying under FLOOD_THRESHOLD (50,000/s) avoids even the flood Warning if the attacker paces the loop. Preconditions: ability to run unprivileged code that issues syscalls (always true for any attacker present on the box) and that on_syscall is actually invoked on the hot path. Caveat that bounds severity: the act cannot suppress the monotonic counters or the dropped tally in the /proc/hunter header, so it leaves a permanent tamper-evident signal; it only destroys the detailed per-event records.

### `WX-6` [LOW] W^X events are advisory-only and trivially evicted from the 512-entry ring, blinding forensics of the very W+X mappings that were allowed
- **Lens:** W^X memory evasion
- **Location:** hunter/src/lib.rs:167-176 (Report records Warning, returns true); hunter/src/event_log.rs:21 (512 ring), 98-107 (push evicts oldest, bumps DROPPED); linux-object/src/fs/procfs.rs (/proc/hunter read-only render)
- **Bypass:** Because WX_MODE=Report, every W+X mmap/mprotect is allowed AND merely appended as a Warning to GLOBAL_LOG, a 512-entry ring (event_log.rs:21). The ring overwrites oldest-first and only increments a DROPPED counter (event_log.rs:98-107). An attacker who creates one W+X mapping (the actual attack) can then issue >512 further benign loggable operations — or simply many more W+X mmaps in a loop — to push the incriminating entry out of the ring; /proc/hunter renders only the surviving 512 entries (lib.rs:200-223, event_log::render). The forensic record of the W^X violation is destroyed by the attacker's own subsequent activity, leaving only an opaque dropped=N count with no per-event detail. Since the W^X check never blocks in default config, the log is the only artifact — and it is attacker-evictable.
- **Exploitability:** Realistic and low-effort under default config. Preconditions: (1) WX_MODE left at its default Report (policy.rs:74) — true out of the box; (2) attacker can run code issuing syscalls — true by default since no per-pid syscall whitelist is registered, so is_syscall_allowed is permissive (policy.rs:158); (3) no external log sink/rotation exists — confirmed, /proc/hunter is read-only in-memory render (procfs.rs:994). The attack is just: do the W+X mapping, then perform >512 further loggable operations (e.g. additional W+X mmaps in a loop) to wrap the 512-entry global ring and evict the per-event detail. No privilege required; the global ring means cross-pid eviction also works. Caveat lowering impact: the monotonic Stats counters (warnings/dropped) and per-entry seq gaps survive eviction, so the fact that W^X warnings occurred and that events were dropped remains visible — only the specific per-event detail (pid/ts/op of the incriminating entry) is lost. It is also not a bypass of any blocking guarantee: in default Report mode the W+X mapping is allowed by policy regardless, and hunter documents WX as audit-only by design.

### `IDS-3` [LOW] Fork-bomb detector is per-parent-pid and resets per process, so a recursive/distributed fork bomb never trips the 200 threshold
- **Lens:** IDS evasion & self-DoS
- **Location:** hunter/src/heuristics.rs:37,145,151,154-164
- **Bypass:** The fork-bomb counter lives in PROC_STATS keyed on the CALLING pid (heuristics.rs:127,151) and counts only clone/fork/vfork issued by that single pid within a 1s window, alerting once at >200 (heuristics.rs:37,161). A classic fork bomb is recursive: each child immediately forks again. So no single pid ever issues 200 forks — the parent forks a handful of children, each child (a fresh pid with its own zeroed ProcStat) forks a handful more, exponentially. The aggregate is a true fork bomb that exhausts the system, yet every individual pid stays far under FORKBOMB_THRESHOLD and the ANOMALY warning never fires. Even a flat fork bomb evades it by spreading forks across a pool of >1 worker pids (each capped at 200), or by recycling pids (a recycled pid starts with a clean ProcStat).
- **Exploitability:** Trivial preconditions. Any unprivileged process able to call clone/fork/vfork (the normal case) evades the alert by using the standard recursive fork-bomb pattern where children re-fork, or a flat bomb spread over >1 worker pid, or by relying on pid recycling. No special privilege, no race, no policy registration needed. The syscall heuristics must be enabled (default on) for the alert to even be eligible to fire, and the alert is advisory only — evasion suppresses an audit log entry; it does not unlock any blocked action, since hunter never blocks forks. Realistic impact is therefore a missing detection signal for an operator/SIEM watching /proc/hunter, not a bypass of any enforcement control or any defense hunter actually implements.

### `IDS-4` [LOW] Rate heuristics alert at most once per window and sit silently above threshold — pace just under 50k/200 to evade entirely; advisory-only anyway
- **Lens:** IDS evasion & self-DoS
- **Location:** hunter/src/heuristics.rs:33-37,90-91,105-113,157-164,167-192
- **Bypass:** Two evasion modes. (a) Stay sub-threshold: flood fires only at syscall_count > 50000 in a 1s window and fork at > 200; an attacker that paces to <=50000 syscalls/sec and <=200 forks/sec is never flagged, which is still a very high abuse rate. (b) Even when crossing, the alert is fire-once-per-window via flood_alerted/fork_alerted flags (heuristics.rs:90-91,157-164): after the single Warning, the process can flood unbounded for the rest of the window with zero further records. And critically the alert is purely advisory — Severity::Warning recorded to the log (heuristics.rs:167-192); there is NO throttle, no kill, no EPERM. So even a detected flood/fork-bomb proceeds at full speed. Combined with IDS-1, the one Warning line can then be evicted from the ring. The window also rolls and resets all counters/flags every 1s (heuristics.rs:105-113), so a burst-then-idle cadence resets the detector each window.
- **Exploitability:** Trivial preconditions: an attacker need only control its own syscall/fork pacing, which any local unprivileged process inherently does. (a) Pacing under 50000 syscalls/sec or 200 forks/sec is fully invisible to the rate heuristics — no code path records anything. (b) When crossing threshold, exactly one Warning line is recorded per 1s window; flooding then proceeds unthrottled because the heuristics are advisory and do not affect check_syscall's verdict (the kernel still runs every permitted syscall). (c) Burst-then-idle across window boundaries re-arms the detector. (d) The single Warning can be aged out of the bounded event ring (per IDS-1) to erase the trace. No bypass of any enforcement gate is involved — the seccomp whitelist, W^X, and exec-path checks are untouched and still apply independently. Real-world impact is limited to reduced/erasable IDS visibility into high-rate abuse, not privilege escalation or control evasion; actual flood/DoS damage depends on the kernel having no other rate limiting, which is outside hunter's claimed scope.

### `LIFE-3` [LOW] Unbounded BTreeMap growth via leaked per-pid state is a memory-exhaustion DoS that also degrades detection
- **Lens:** race / TOCTOU / lifecycle
- **Location:** hunter/src/heuristics.rs:117,151,196; hunter/src/policy.rs:24,133; linux-syscall/src/task.rs:650
- **Bypass:** PROC_STATS (heuristics.rs:117) gains an entry on the first syscall of every PID (heuristics.rs:151 stats.entry(pid).or_insert_with) and is freed only by forget(), which is only reachable through task_exit → only wired to sys_exit_group (task.rs:650). An attacker spawns a large number of short-lived processes that each terminate via a fatal signal or via exit(2) (never exit_group). Each such process permanently leaks a ProcStat into the global BTreeMap (and, if any whitelist registration existed, a GLOBAL_POLICIES Vec too). The map grows without bound, consuming kernel heap, and since it is a single Mutex-protected global BTreeMap consulted under lock on every single syscall (heuristics.rs:150 PROC_STATS.lock()), its growth also increases lookup cost on the syscall hot path for every process system-wide — a self-amplifying DoS. No eviction, no cap, no TTL exists.
- **Exploitability:** Reachable by any unprivileged process able to spawn children: terminate each child via a raw _exit(2)/sys_exit (single-thread group) or by sending it a fatal signal/SIGKILL, both of which skip sys_exit_group and thus hunter::task_exit. Each leaked entry is a small ProcStat (a handful of u64/bool fields) plus a BTreeMap node — tens of bytes — so meaningful heap exhaustion requires creating on the order of hundreds of thousands to millions of such processes, i.e. sustained deliberate churn rather than a one-shot crash. No special privilege, config, or race window needed, and it is not blocked by the surrounding kernel. Note that normal libc-linked workloads exit via exit_group and are correctly reaped, so this is an attacker-driven slow leak, not a passive system-wide one. The alarming sub-claims (detection bypass via stale state, self-amplifying hot-path slowdown) are not realizable here, which is why the corrected severity is low.

### `EXEC-1` [LOW] execve gate is magic-only over 64 bytes with substring path matching and Report default
- **Lens:** threat-model & info-leak
- **Location:** hunter/src/lib.rs:106-146; hunter/src/policy.rs:75,174-180; linux-syscall/src/task.rs:547-551
- **Bypass:** check_elf_binary 'integrity' is only a 4-byte \x7fELF magic compare over the first 64 bytes read (task.rs:547, lib.rs:108) -- no hash/signature/section validation, so any genuine ELF payload passes. The path policy (the only substantive gate) defaults to EXEC_MODE=Report (policy.rs:75) and is never changed, so exec from /tmp, /var/tmp, /dev/shm is logged-and-allowed. Even under Enforce the check is pure string matching: is_untrusted_exec_path uses starts_with over a fixed 3-prefix list plus a substring ../ test (policy.rs:174-180) with no path canonicalization. The attacker stages the payload outside those prefixes (/run, /home, /mnt), or uses a symlink under /bin pointing into /tmp, a bind-mount, or a hardlink -- the path string handed to execve never starts with a blocked prefix, so it executes.
- **Exploitability:** Trivial in the default config (exec=Report): every code read in the finding is accurate. EXEC_MODE is AtomicU8::new(1)=Report at policy.rs:75 and set_exec_mode(Enforce) is reachable only from hunter's own command handler (lib.rs:262) / tests, never at boot, so exec from /tmp, /var/tmp, /dev/shm is logged-and-allowed by default. Even after an operator opts into Enforce, the path gate is bypassable as designed: is_untrusted_exec_path (policy.rs:174-179) blocklists only 3 hardcoded prefixes and matches the RAW, un-canonicalized execve argument (task.rs:549 passes path_str directly; lookup_inode follows symlinks for the inode but the checked STRING is pre-resolution). So: (1) stage the ELF anywhere outside the 3 prefixes (/run, /home, /mnt, cwd-relative) — never matches starts_with; (2) symlink /bin/x -> /tmp/payload and exec /bin/x — the string '/bin/x' never starts with a blocked prefix; (3) bind-mount or hardlink achieve the same. The ../ test is pure substring, so only literal '/bin/../tmp/x' is caught, not symlink/abs-path equivalents. Precondition: an unprivileged user with any writable location outside the 3 prefixes or the ability to create a symlink/hardlink/bind-mount — standard.

### `STATE-1` [LOW] Cleanup wired only to exit_group; pid recycling, fork-topology, and advisory-only alerts evade/abuse anomaly state
- **Lens:** threat-model & info-leak
- **Location:** hunter/src/lib.rs:193-196; linux-syscall/src/task.rs:650; hunter/src/heuristics.rs:145-164,167-192,196-198
- **Bypass:** task_exit (which removes policy + anomaly counters) is called only from sys_exit_group (task.rs:650). Processes that terminate via plain exit, signal kill, or thread teardown never call it, leaking GLOBAL_POLICIES/PROC_STATS entries; an attacker spawns and kills many short-lived processes (non-exit_group) to grow these BTreeMaps unbounded (memory DoS). The fork-bomb counter is per-parent-pid (heuristics.rs:161): a recursive fork bomb where each child itself forks attributes counts to fresh child pids, each staying below the 200/window threshold, so it never trips. And every anomaly alert is Warning-only with no throttle or kill (heuristics.rs:167-192) -- purely advisory -- so even a detected flood/fork-bomb takes no action and the log entry can then be evicted per EVADE-1.
- **Exploitability:** Standard unprivileged userspace. PROC_STATS leak: any process that makes >=1 syscall and then terminates via sys_exit (_exit), signal, or kill — i.e. the common case, since glibc _exit/single-threaded exit routes through sys_exit not exit_group — leaves a permanent ProcStat entry; loop spawn+exit to grow the BTreeMap toward memory exhaustion (slow: one small entry per process, so practical DoS needs very many iterations). Fork-bomb evasion: launch a recursive fork bomb (each child forks a few times) rather than one process looping clone; per-pid counts stay under 200/window so no fork-bomb event is logged. Neither bypasses hunter's actual enforcement (syscall whitelist / W^X / exec-path), and the advisory-only alerting is documented intended behavior, not a defeated control.

## Prioritized hardening plan

### P1 — Close the file-backed mmap RXW ceiling: map with the requested prot, not hardcoded MMUFlags::RXW  _(hooks, effort S)_
- **Addresses:** WX-3, SYS-4, SYS-8, WX-1
- **Change:** In linux-syscall/src/vm.rs:134-143 replace the hardcoded `permissions=MMUFlags::RXW` argument to `vmar.map_ext` with a ceiling derived from the actual requested prot, e.g. `prot.to_flags() | MMUFlags::USER` (read+user always, write/exec only if requested). This makes the per-page protection ceiling (is_valid_mapping_flags, vmar.rs:931-933) actually constrain later mprotect transitions instead of permitting any RWX flip. Keep map_range=false to preserve demand-paging. If a known JIT/loader regresses, special-case it behind WX_MODE==Off rather than re-widening the ceiling for everyone.
- **Rationale:** This single line is the load-bearing structural W^X bypass: even a fully-enforcing hunter cannot stop exec-on-writable-file-pages while the kernel itself installs an RXW ceiling. Tightening the ceiling is what gives every later W^X check teeth.
- **Breakage risk:** Medium: a dynamic loader that maps a file PROT_READ and later mprotects it to add WRITE (rare for file-backed) would now get EINVAL from the ceiling check. Mitigate by mapping file regions with READ|WRITE ceiling when MAP_PRIVATE (COW) is requested, but never adding EXECUTE unless PROT_EXEC was in the original prot. Land behind a feature/mode so boot can be validated first.

### P2 — Make mprotect tightening authoritative: stop swallowing protect() errors as Ok(0)  _(hooks, effort M)_
- **Addresses:** SYS-8, WX-3, SYS-4
- **Change:** In linux-syscall/src/vm.rs:298-307, when WX_MODE==Enforce do not return Ok(0) on `vmar.protect` Err. Either propagate the error (return LxError::EINVAL/ENOMEM) or split the requested range to sub-region boundaries and re-issue protect per sub-region so the narrowing actually applies. The current 'benign no-op' is only acceptable when WX is Off/Report; under Enforce a failed narrowing that leaves pages executable+writable must fail the syscall. Gate the lenient fallback on `hunter::policy::wx_mode() != Mode::Enforce`.
- **Rationale:** Pairs with item 1: even with a correct ceiling, a defender cannot rely on mprotect ever tightening permissions while any Err silently reports success, so an attacker keeps the liberal mapping and the syscall claims it succeeded.
- **Breakage risk:** Medium: some legitimate range-spanning mprotects rely on the no-op fallback today; under Report (the default) behavior is unchanged, so boot is unaffected. Only operators who raise WX to Enforce see stricter behavior, which is the intended opt-in contract.

### P3 — Track per-region W-then-X history so the mmap-W / mprotect-X two-step is detectable  _(hooks, effort L)_
- **Addresses:** WX-2, WX-1, SYS-4
- **Change:** Add a per-process interval map of ever-writable regions in hunter (new module hunter/src/wx.rs, keyed pid -> sorted Vec<(start,end)> of regions that were ever mapped/mprotected WRITE). Extend check_mmap/check_mprotect signatures to take (addr,len). On any mapping that grants WRITE, record the interval; on any mprotect/mmap that grants EXEC, if the target range intersects a previously-writable interval, treat it as a W^X violation (block in Enforce, warn in Report). Clear intervals for a region on munmap and for a pid on task_exit. Update call sites vm.rs:106 and vm.rs:285 to pass addr/len; add a check_munmap hook.
- **Rationale:** Without region history hunter only ever sees per-call boolean flags, so the canonical JIT/shellcode flip (write then exec in separate calls) is invisible even under Enforce. This is the core IDS gap for W^X.
- **Breakage risk:** Medium-High: JITs legitimately do W-then-X on the same region; in Enforce this would break them. Keep WX default Report so the two-step only logs by default, and document that Enforce requires JIT-aware allowlisting. The interval map adds locked state on the mmap/mprotect path (not the syscall hot path), bounded and freed at exit, so DoS surface is limited.

### P4 — Provide a process-creation hook that registers a syscall whitelist and re-keys state at fork/exec  _(hooks, effort M)_
- **Addresses:** SYS-1, SYS-5, LIFE-2, STATE-1
- **Change:** Add hunter::task_fork(parent_pid, child_pid) and hunter::task_exec(pid, path) hooks. Wire task_fork into the clone/fork/vfork path (linux-syscall task creation) to optionally inherit the parent's whitelist into the child (so a process cannot shed policy by forking) and to seed a fresh ProcStat. Wire task_exec into sys_execve (task.rs, after the ELF check) to: clear/re-evaluate the per-pid anomaly window (prevent benign-then-malicious exec laundering) and re-apply any default whitelist for the new image. Provide a kernel-callable policy::install_default_policy() invoked at boot for init so active_policy_count() is non-zero and the seccomp path is exercised. Keep the default whitelist empty/absent unless an operator opts in, preserving permissive boot.
- **Rationale:** The seccomp layer is permanently open because there is no creation-time hook to ever populate a whitelist, and state never follows process identity across fork/exec. A fork/exec hook is the canonical seccomp re-evaluation point and the only way to make the whitelist anything other than dead code.
- **Breakage risk:** Low if defaults stay permissive (no whitelist registered => None branch => Ok). Risk rises only when an operator registers policies; inheritance-at-fork must include the syscalls a child needs (clone/execve/exit) or it breaks process creation. Ship with whitelist absent by default; the hook is plumbing, not enforcement.

### P5 — Free per-pid state on every process teardown path, not just exit_group; add a hard cap/TTL  _(hooks, effort M)_
- **Addresses:** SYS-2, IDS-2, LIFE-3, STATE-1
- **Change:** Call hunter::task_exit from every terminal path: sys_exit when it is the last thread of the group (task.rs:634-638), the signal-kill/zircon process-exit path, and thread-group teardown. Centralize by calling task_exit from the process's exit callback (the same place zircon_process().exit runs) rather than only the syscall wrappers. Additionally, in heuristics.rs and policy.rs bound the BTreeMaps: enforce a MAX_TRACKED_PIDS cap and evict the least-recently-updated ProcStat when exceeded, and skip lazily creating a ProcStat for pids that only ever issue non-sensitive syscalls if over cap. This guarantees freeing even if a teardown path is missed.
- **Rationale:** Today an attacker spawning many _exit/signal-killed processes leaks BTreeMap entries forever under a single Mutex taken on every syscall, an unprivileged kernel-heap DoS that also slows the whole machine's syscall path. Hooking all teardown paths plus a hard cap makes the leak and DoS impossible.
- **Breakage risk:** Low: freeing state for a dead pid is always correct. The cap could evict a live process's anomaly window under extreme churn (degrading detection for that pid) but never breaks userspace. Centralizing on the process exit callback avoids missing future teardown paths.

### P6 — Resolve and canonicalize the exec path before policy evaluation; check the loaded inode, not the user string  _(hooks, effort M)_
- **Addresses:** ELF-2, ELF-3, EXEC-1, SYS-3, SYS-4(critic)
- **Change:** Change the execve hook so hunter evaluates the *resolved* target, not the raw argument. In task.rs:540-552, after lookup_inode resolves the path (following symlinks and /proc/self/fd/N magic-links, mod.rs:1170-1225), pass the canonical absolute path (and ideally the resolved inode's fs path) to check_elf_binary instead of path_str. Implement canonicalization (strip ./, resolve .. against cwd, reject non-absolute after resolution) before is_untrusted_exec_path. For the magic-link/relative cases, derive the path from the resolved inode rather than trusting the literal execve argument.
- **Rationale:** The path policy checks a string that is not what gets executed: /proc/self/fd/N, symlinks, and relative spellings all resolve to a blocked file while presenting a trusted-looking string, defeating the gate even under Enforce. Evaluating the resolved target is the only correct fix.
- **Breakage risk:** Low-Medium: canonicalization must match the loader's own resolution exactly or a legitimate exec could be misclassified. Since EXEC default is Report, mismatches only over-log by default. Requires threading the resolved path/inode from the lookup into the hook.

### P7 — Audit the dynamic linker and shebang interpreter through the same exec gate  _(hooks, effort M)_
- **Addresses:** ELF-1, SYS-4(critic)
- **Change:** In the loader (loader/mod.rs around the shebang resolution at :126/:152 and the PT_INTERP resolution at :172/:213-255), call hunter::check_elf_binary (or a new hunter::check_interp) on each interpreter inode before mapping it, passing the resolved interpreter path and its first bytes. This closes the `#!/tmp/evil` and `/tmp/ld.so` chains that currently execute with zero hunter visibility. Respect MAX_INTERP_DEPTH and record each hop in the event log so the interpreter chain is in the forensic trail.
- **Rationale:** An attacker drops a trusted-looking launcher whose interpreter is attacker-controlled; the launcher passes hunter while the loader recursively executes the malicious interpreter unchecked. Even the magic check is skipped for shebang interpreters.
- **Breakage risk:** Medium: the dynamic linker (/lib/ld-musl) lives in a trusted path so normal dynamic binaries pass; mis-set untrusted prefixes could block ld.so under Enforce. Default Report avoids breakage; treat as audit-first.

### P8 — Unify and strengthen the second (loader) exec gate; read enough bytes and canonicalize args[0]  _(hooks, effort S)_
- **Addresses:** SYS-3(critic), EXEC-1, ELF-4
- **Change:** In loader/src/linux.rs:160-166, replace the 4-byte `[0u8;4]` read with the same 64-byte (or larger) header read used in task.rs:547, and resolve args[0] to an absolute canonical path before calling check_elf_binary so the boot/spawn path applies the same policy as runtime execve. Factor the header-read + canonicalize + check into a single hunter::check_exec(inode, path) helper called by both call sites to eliminate divergence.
- **Rationale:** There are two exec gates with different strength; the earlier loader gate reads only 4 bytes against a bare name like 'busybox', so boot/initial processes get the weakest check and the prefix matching is meaningless on a non-canonical name.
- **Breakage risk:** Low: boot processes come from trusted paths and Report default means no blocking; the change is mostly making the two paths consistent. Reading 64 bytes from the boot VMO is cheap.

### P9 — Strengthen ELF integrity beyond 4-byte magic and close the header-vs-VMO TOCTOU  _(hooks, effort M)_
- **Addresses:** ELF-4, EXEC-1, SYS-4(critic)
- **Change:** In check_elf_binary (lib.rs:106-118) validate e_ident class/data/version, e_type (ET_EXEC/ET_DYN), and e_machine against the build target so a blob that is merely magic-prefixed but malformed/foreign is rejected. Crucially, eliminate the double-read TOCTOU: in task.rs the integrity check reads header via inode.read_at (548) while execution loads a separate read_as_vmo (553). Validate the SAME bytes that are mapped — derive the header from the loaded VMO (read the first page of `vmo`) or hold a read reference so a racing writer/remount cannot swap the file between the two reads. Long term, add an optional signature/hash domain (verify against an allowlist of known-good hashes) behind a new EXEC integrity mode, default Off.
- **Rationale:** The only always-on exec guarantee is a 4-byte compare any ELF satisfies, and even that inspects different bytes than are executed (a racing-writer / magic-link TOCTOU). Validating the loaded bytes and real ELF fields raises the floor meaningfully.
- **Breakage risk:** Low-Medium: stricter e_machine/e_type checks could reject unusual-but-valid binaries; gate the stricter checks so only obviously-foreign images fail. Reading the header from the VMO is a pure correctness improvement with no userspace impact.

### P10 — Make the forensic log tamper-evident: pin high-severity entries and add a severity-segregated reserve  _(event_log, effort M)_
- **Addresses:** IDS-1, EVADE-1, WX-6, SYS-6(uncertain), SYS-10(critic)
- **Change:** In hunter/src/event_log.rs split GLOBAL_LOG into two rings (or one ring with a protected sub-region): a large evictable ring for Info/Notice and a separate, non-evicting (or much slower-evicting) ring reserved for Warning/Critical so flood of benign Notice events cannot evict attack evidence. When the Critical ring would overflow, prefer dropping oldest *Notice* first. Additionally, make eviction visible per-severity: track dropped_by_severity and surface 'critical_dropped=N' in render_report so an operator cannot miss that high-severity events were lost. Fix the BLOCKED counter (event_log.rs:160) to also count Enforce-blocked actions regardless of the exact action string, and add a 'warnings_allowed' (Report-mode) counter so render_report does not misleadingly show blocked=0 while attacks proceed in Report.
- **Rationale:** The log is the only artifact when checks run in Report (the default), and it is fully attacker-evictable with ~512 cheap events; the headline counters also under-report Report-mode-allowed attacks (blocked=0 looks safe). Pinning Critical/Warning and exposing critical-drops makes forensics survivable.
- **Breakage risk:** Low: more memory for the reserved ring; no userspace-visible behavior change beyond a richer /proc/hunter. No boot impact.

### P11 — Add an off-box / persistent sink hook and rate-limit self-generated WATCH events  _(event_log, effort M)_
- **Addresses:** IDS-1, EVADE-1, SYS-6(uncertain)
- **Change:** Add an optional callback (hunter::event_log::set_sink(fn(&LogEntry))) the kernel can wire to a serial/console/persistent sink so Warning+/Critical events are emitted off the volatile ring before any eviction can erase them. Separately, de-duplicate/rate-limit the per-syscall WATCH records in heuristics.rs:130-138 (currently un-throttled, unlike the flood/fork alerts) so an attacker cannot use his own WATCH-generating syscalls as cheap ring-filler. Coalesce repeated identical WATCH events per (pid,category) within a window into a count.
- **Rationale:** Even with pinning (item 10), a determined attacker can flood Warning-class events; an append-only off-box sink defeats in-ring eviction entirely, and rate-limiting WATCH removes the cheapest flooding primitive.
- **Breakage risk:** Low: the sink is opt-in and called outside locks; rate-limiting WATCH could merge distinct sensitive calls into one line (mitigated by including a count). No boot/userspace impact.

### P12 — Replace the unauthenticated transmute'd clock with a validated, freeze-resistant time source  _(boot, effort M)_
- **Addresses:** SYS-1(critic), SYS-2(critic)
- **Change:** In hunter/src/clock.rs stop transmuting an AtomicUsize into a fn pointer on every now_ns(). Use a typed `static TIME_SOURCE: AtomicPtr<()>` set once and validated, or better, gate set_time_source so it can only be called once (a 'sealed' flag) and refuse a null/0 store. Detect a frozen or non-monotonic clock in now_ns()/ProcStat::roll: if successive reads do not advance, fall back to a monotonic event-sequence counter for windowing instead of wall-clock so the IDS window cannot be silenced by a 0-returning or jumping clock. Make the IDS sliding window count-bounded as a backstop (roll after N events even if the clock has not advanced) so flood/fork detection cannot be structurally disabled by clock behavior.
- **Rationale:** The clock is the single shared dependency of both forensics and the IDS, set via an ungated transmute target; set_time_source(0) freezes all timestamps and a fast/jumping clock rolls the window every syscall so flood/fork-bomb can never accumulate. A count-bounded backstop makes detection independent of an untrusted clock.
- **Breakage risk:** Low: sealing set_time_source is compatible with the single boot-time call (main.rs:65); the count-bounded window only adds a second trigger condition and never blocks userspace.

### P13 — Harden the control plane: authorize, log, and (optionally) seal all mode/policy mutators  _(policy, effort M)_
- **Addresses:** SYS-9(critic), SYS-6(uncertain), IDS-6(uncertain)
- **Change:** Wrap every mutator (set_syscall_mode/set_wx_mode/set_exec_mode in policy.rs:86-110, register_policy, add_untrusted_exec_prefix, heuristics::set_anomaly_detection) so each transition records an Info/Notice event with old->new value (today set_* records nothing). Add a one-way 'tighten-only' or 'sealed-after-boot' option: once an operator raises a domain to Enforce and seals, further relaxation to Off/Report is refused (or requires a capability), so a single relaxed store cannot silently neutralize hunter. Use Ordering::SeqCst (not Relaxed) for mode loads/stores that gate enforcement to avoid racy reads. Keep modes global for now but document the per-process limitation.
- **Rationale:** Every mode is a global, relaxed, unauthenticated, unlogged AtomicU8; any reachable setter (or a future /proc handler exposing one) flips the whole machine to Off with zero forensic trace. Logging transitions and sealing make the control plane tamper-evident and tamper-resistant.
- **Breakage risk:** Low: logging transitions is harmless; sealing is opt-in and only restricts relaxation, never normal operation. SeqCst on a cold path has negligible cost.

### P14 — Make the sensitive-syscall and fork tables arch-correct via a per-arch syscall mapping  _(heuristics, effort M)_
- **Addresses:** SYS-3(uncertain), SYS-7(critic), IDS-6(uncertain), IDS-3
- **Change:** Replace the hardcoded x86_64 constants in heuristics.rs:40-62 with a per-arch table selected by cfg(target_arch) (x86_64/riscv64/aarch64), or have the syscall layer pass a normalized logical syscall id to hunter so classify()/is_fork match the right operations on every target. Critically this also fixes the fork-bomb signal (SYS_CLONE/FORK/VFORK 56/57/58 are x86_64-only) so detection works on riscv64/aarch64. Add an aggregate/cross-pid fork-bomb signal (track total system-wide clone rate, or attribute child forks up to the process-group/ancestor) so a recursive fork bomb where each child forks once still trips a threshold.
- **Rationale:** On non-x86_64 targets the watch flags the wrong syscalls and the fork-bomb detector counts the wrong numbers, silently disabling detection on supported arches; and even on x86_64 the per-parent-pid fork-bomb counter never trips for a recursive bomb. Arch-correct tables plus an aggregate signal make the IDS actually fire.
- **Breakage risk:** Low: advisory-only signals; wrong-arch fixes can only improve fidelity. The aggregate fork-bomb counter must be tuned to avoid false positives on legitimate process-heavy workloads, but it never blocks (advisory).

### P15 — Add optional active throttling/denial for detected floods, fork bombs, and privileged syscalls  _(heuristics, effort L)_
- **Addresses:** IDS-4, SYS-6(critic), SYS-3(uncertain), STATE-1
- **Change:** Give the IDS an Enforce capability: when ANOMALY mode is Enforce (new per-domain mode, default Report), a confirmed flood/fork-bomb returns a deny that the syscall dispatcher honors (EAGAIN/EPERM) or throttles, instead of recording a single advisory Warning and proceeding. Route the privileged-syscall class (init_module/finit_module/bpf/kexec/ptrace/mount/pivot_root/setns) through is_syscall_allowed (or a dedicated deny list) so they can be blocked under Enforce rather than only watched; for ptrace also inspect the target pid argument. Keep all of this default-Report so boot/userspace are unaffected until an operator opts in.
- **Rationale:** Every anomaly and every dangerous syscall is currently advisory-only with no throttle/kill/EPERM, so even a detected fork bomb or a kernel-module load proceeds at full speed; the most dangerous operations are explicitly demoted to advisory. An opt-in Enforce path is the only way detection can prevent rather than merely narrate.
- **Breakage risk:** Medium when enabled: denying privileged syscalls or throttling can break legitimate admin tools; hence default Report and per-domain opt-in. Wiring a deny back into the dispatcher is the highest-effort integration but reuses the existing check_syscall return contract (lib.rs:130 already maps Err->EPERM).

## Residual risks

- Global, not per-process, modes: even after authorization/logging/sealing (item 13), enforcement remains a single global stance. A future per-process or per-container policy model is needed for true isolation; until then one sealed-Off mistake or a capability compromise still affects the whole machine.
- W^X Enforce remains incompatible with JITs/dynamic loaders that legitimately do W-then-X (item 3). Real enforcement requires per-application allowlisting or a PROT-based JIT contract; shipping Enforce broadly will break such userspace, so it stays opt-in and most systems will run W^X in Report indefinitely.
- Exec integrity without code signing is still bypassable by any attacker who can write a valid ELF to a trusted path (e.g. compromising a process that can write under /bin or a trusted prefix). The hash/signature domain (item 9) is optional and default-Off; until populated with a trusted hash set, integrity is structural-validity only, not authenticity.
- Path canonicalization (item 6) must exactly mirror the VFS/loader resolution including mounts, bind mounts, overlay/remount, and FOLLOW_MAX_DEPTH; any divergence reintroduces a string-vs-inode bypass. Bind/overlay remounts that change what a canonical path resolves to remain a TOCTOU surface between policy evaluation and load.
- The forensic log, even with pinned Critical entries and an off-box sink (items 10-11), can be overwhelmed if the sink is not wired or is itself rate-limited; a sufficiently large Critical flood still forces eviction within the reserved ring. Durable, append-only off-box storage outside attacker reach is the only complete answer and is out of scope for the in-kernel crate.
- Anomaly thresholds (flood 50k/s, fork 200/s, and any new aggregate) are inherently evadable by pacing just under them; raising fidelity reduces but never eliminates low-and-slow abuse, and tightening risks false positives on legitimate high-throughput workloads.
- The clock hardening (item 12) reduces but does not remove trust in the platform timer; a compromised timer HAL can still feed plausible-but-wrong timestamps. The count-bounded window protects detection, but forensic timestamps remain only as trustworthy as kernel-hal.
- pid-0 / low-pid early-boot and kernel-spawned contexts still share the pid namespace used for keying and for the event_log 'pid=0 means kernel' convention, so attacker events under a low pid can be confused with subsystem events unless a distinct identity/tag for kernel-origin events is added.
- Adding hooks at fork/exec/teardown and per-region W^X tracking increases locked state and work on hot creation/mapping paths; under extreme churn this is itself a (bounded) performance cost, and the MAX_TRACKED_PIDS cap trades detection completeness for memory safety, so a high-churn attacker can still evict live anomaly windows (degraded detection, not DoS).
